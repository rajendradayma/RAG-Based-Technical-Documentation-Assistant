# FAISS Documentation

FAISS (Facebook AI Similarity Search) is a library for efficient similarity search and clustering of dense vectors. It contains algorithms that search in sets of vectors of any size, up to ones that possibly do not fit in RAM. It is developed by Meta's Fundamental AI Research team.

Key features include:
- **Speed**: Optimized C++ implementation with GPU support for billions of vectors.
- **Memory efficiency**: Compressed representations reduce memory usage by 10-100x.
- **Scalability**: Handles datasets from thousands to billions of vectors.
- **Flexibility**: Supports exact and approximate search, GPU acceleration, and distributed search.
- **Integration**: Works seamlessly with NumPy, PyTorch, and popular embedding models.

FAISS is particularly well-suited for:
- Large-scale semantic search and retrieval.
- Recommendation systems.
- Image and video similarity search.
- Clustering and nearest neighbor search.
- RAG (Retrieval-Augmented Generation) pipelines.

## Core Concepts

FAISS is built around a few fundamental concepts that are essential to understand before using the library.

### Vectors and Embeddings

FAISS operates on dense vectors — arrays of floating-point numbers that represent data in a high-dimensional space. These vectors are typically generated by embedding models (e.g., sentence transformers, image encoders, or language models).

**Key properties:**
- **Dimensionality (d)**: The length of each vector (e.g., 384, 768, 1024).
- **Dataset size (n)**: The number of vectors in your index.
- **Distance metric**: How similarity is measured (L2, inner product, cosine).

### Index

An index in FAISS is a data structure that stores vectors and enables fast similarity search. Different index types offer different trade-offs between search speed, memory usage, and accuracy.

**Index lifecycle:**
1. **Create**: Initialize the index with the vector dimensionality.
2. **Add**: Insert vectors into the index.
3. **Search**: Query the index for nearest neighbors.
4. **Save/Load**: Persist the index to disk or load it back.

### Distance Metrics

FAISS supports several distance metrics:

| Metric | Description | Best For |
|--------|-------------|----------|
| **L2 (Euclidean)** | Straight-line distance between vectors | General-purpose similarity |
| **Inner Product** | Dot product of vectors | Maximum inner product search (MIPS) |
| **Cosine** | Angle between vectors (requires normalization) | Text embeddings, semantic search |

**Important**: For cosine similarity, normalize your vectors to unit length before adding them, then use inner product search.

## Creating an Index

The simplest FAISS index performs exact brute-force search. This is suitable for small datasets (up to ~100K vectors).

### Exact Search with IndexFlatL2

```python
import faiss
import numpy as np

# Vector dimensionality
d = 128

# Create a flat L2 index
index = faiss.IndexFlatL2(d)

print(f"Index is trained: {index.is_trained}")  # True for flat indexes
```

**IndexFlatL2 characteristics:**
- **Exact search**: Returns the true k-nearest neighbors.
- **No training required**: Ready to use immediately.
- **Memory usage**: Stores vectors in full precision (4 * d bytes per vector).
- **Search speed**: O(n * d) — linear in dataset size.
- **Best for**: Datasets up to ~100K vectors where exact results are required.

### Adding Vectors

```python
# Generate random vectors (in practice, these are your embeddings)
np.random.seed(42)
vectors = np.random.random((10000, d)).astype('float32')

# Add vectors to the index
index.add(vectors)

print(f"Total vectors in index: {index.ntotal}")
```

**Important notes about adding vectors:**
- Vectors must be a NumPy array of type `float32`.
- The array shape must be `(n, d)` where `n` is the number of vectors and `d` is the dimensionality.
- You can add vectors in batches to manage memory.
- Once added, vectors cannot be removed from most index types (use `IndexIDMap` for deletable indexes).

## Searching

### Basic Search

```python
# Number of nearest neighbors to retrieve
k = 5

# Query vector (must be float32)
query = np.random.random((1, d)).astype('float32')

# Search
distances, indices = index.search(query, k)

print(f"Nearest neighbors: {indices[0]}")
print(f"Distances: {distances[0]}")
```

**Search output:**
- `distances`: Array of shape `(n_queries, k)` containing the distances to nearest neighbors.
- `indices`: Array of shape `(n_queries, k)` containing the indices of nearest neighbors in the original dataset.

### Batch Search

```python
# Multiple queries at once
queries = np.random.random((100, d)).astype('float32')
distances, indices = index.search(queries, k)

print(f"Results shape: {indices.shape}")  # (100, 5)
```

Batch search is significantly more efficient than searching one vector at a time because it amortizes overhead.

## Approximate Search with IVF

For larger datasets, exact search becomes too slow. FAISS provides approximate search indexes that trade a small amount of accuracy for massive speedups.

### Inverted File Index (IVF)

IVF indexes partition the vector space into clusters (Voronoi cells) and only search the most relevant clusters.

```python
# Number of clusters (list size)
nlist = 100

# Create IVF index with flat quantizer
quantizer = faiss.IndexFlatL2(d)
index_ivf = faiss.IndexIVFFlat(quantizer, d, nlist)

# Train the index (required for IVF)
index_ivf.train(vectors)

# Add vectors
index_ivf.add(vectors)

# Search
index_ivf.nprobe = 10  # Number of clusters to search
distances, indices = index_ivf.search(query, k)
```

**IVF parameters:**
- `nlist`: Number of clusters. Higher = more accurate but slower. Rule of thumb: `sqrt(n)`.
- `nprobe`: Number of clusters to search per query. Higher = more accurate but slower.

**IVF trade-offs:**
- **Speed**: 10-100x faster than exact search.
- **Memory**: Same as flat index (stores full vectors).
- **Accuracy**: Near-exact with proper `nprobe` tuning.
- **Training required**: Yes, must call `train()` before `add()`.

## Product Quantization (PQ)

For very large datasets where memory is a constraint, Product Quantization compresses vectors into compact codes.

### IndexIVFPQ

```python
# Number of sub-quantizers
m = 16  # d must be divisible by m

# Bits per sub-quantizer
nbits = 8

# Create IVF-PQ index
quantizer = faiss.IndexFlatL2(d)
index_pq = faiss.IndexIVFPQ(quantizer, d, nlist, m, nbits)

# Train and add
index_pq.train(vectors)
index_pq.add(vectors)

# Search
index_pq.nprobe = 10
distances, indices = index_pq.search(query, k)
```

**PQ compression explained:**
- Each vector is split into `m` sub-vectors.
- Each sub-vector is quantized to `nbits` bits.
- Total compression: `d * 32` bits → `m * nbits` bits.
- Example: 128-dim float32 vector (512 bytes) → 16 * 8 = 128 bits (16 bytes) = 32x compression.

**PQ trade-offs:**
- **Memory**: 10-50x reduction compared to flat indexes.
- **Speed**: Fast search, but slower than IVF-Flat.
- **Accuracy**: Lower than IVF-Flat; requires careful parameter tuning.
- **Training**: Requires representative training data.

## HNSW Index

Hierarchical Navigable Small World (HNSW) graphs provide state-of-the-art approximate nearest neighbor search with excellent speed-accuracy trade-offs.

### IndexHNSWFlat

```python
# Create HNSW index
index_hnsw = faiss.IndexHNSWFlat(d, 32)  # 32 = M parameter

# Add vectors (no training needed)
index_hnsw.add(vectors)

# Search
index_hnsw.hnsw.efSearch = 64  # Search-time accuracy parameter
distances, indices = index_hnsw.search(query, k)
```

**HNSW parameters:**
- `M`: Number of neighbors per node in the graph. Higher = more accurate, more memory. Typical: 16-64.
- `efConstruction`: Build-time accuracy. Higher = better graph quality. Typical: 40-200.
- `efSearch`: Search-time accuracy. Higher = more accurate, slower. Typical: 16-200.

**HNSW characteristics:**
- **Speed**: Extremely fast, often the fastest for in-memory search.
- **Memory**: Moderate overhead from graph structure.
- **Accuracy**: Excellent with proper parameter tuning.
- **Incremental**: Supports adding vectors after creation.

## Saving and Loading Indexes

FAISS indexes can be persisted to disk and loaded back.

### Saving an Index

```python
# Save index to file
faiss.write_index(index, "my_index.faiss")
```

### Loading an Index

```python
# Load index from file
index = faiss.read_index("my_index.faiss")

print(f"Loaded index with {index.ntotal} vectors")
```

**Best practices for persistence:**
- Save indexes after training and adding all vectors.
- Use `write_index`/`read_index` for single indexes.
- For distributed systems, consider `faiss.write_index_binary` for compressed indexes.
- Store metadata (original text, IDs) separately — FAISS only stores vectors and indices.

## Working with IDs

By default, FAISS returns the position (0-based index) of vectors in the added order. To use custom IDs, wrap the index with `IndexIDMap`.

### Using Custom IDs

```python
# Create a flat index with ID mapping
index_flat = faiss.IndexFlatL2(d)
index_with_ids = faiss.IndexIDMap(index_flat)

# Add vectors with custom IDs
ids = np.arange(10000, 20000, dtype='int64')
index_with_ids.add_with_ids(vectors, ids)

# Search returns custom IDs
distances, indices = index_with_ids.search(query, k)
print(f"Custom IDs: {indices[0]}")  # e.g., [10042, 10123, ...]
```

**When to use ID mapping:**
- When you need to map search results back to original data.
- When vector positions don't correspond to meaningful identifiers.
- When you need to remove vectors by ID (requires `IndexIDMap2`).

## GPU Acceleration

FAISS provides GPU implementations that can dramatically speed up search and training.

### Moving an Index to GPU

```python
# Get the default GPU resource
res = faiss.StandardGpuResources()

# Move index to GPU
gpu_index = faiss.index_cpu_to_gpu(res, 0, index)

# Search on GPU
distances, indices = gpu_index.search(query, k)
```

**GPU considerations:**
- GPU memory limits the maximum index size.
- Multiple GPUs can be used with `index_cpu_to_all_gpus`.
- Not all index types support GPU (check FAISS documentation).
- Data transfer between CPU and GPU adds overhead for small batches.

## Integration with Embedding Models

FAISS is typically used alongside embedding models in RAG pipelines.

### Example with Sentence Transformers

```python
from sentence_transformers import SentenceTransformer
import faiss
import numpy as np

# Load embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')

# Your documents
documents = [
    "FastAPI is a modern web framework",
    "Pydantic validates data with type hints",
    "LangGraph builds stateful LLM applications",
    "FAISS enables efficient similarity search"
]

# Generate embeddings
document_embeddings = model.encode(documents, convert_to_numpy=True)
document_embeddings = document_embeddings.astype('float32')

# Create and populate FAISS index
d = document_embeddings.shape[1]
index = faiss.IndexFlatL2(d)
index.add(document_embeddings)

# Search
query = "How do I build LLM apps?"
query_embedding = model.encode([query], convert_to_numpy=True).astype('float32')

k = 2
distances, indices = index.search(query_embedding, k)

for i, idx in enumerate(indices[0]):
    print(f"Result {i+1}: {documents[idx]} (distance: {distances[0][i]:.4f})")
```

**Integration best practices:**
- Normalize embeddings for cosine similarity (use `IndexFlatIP` with normalized vectors).
- Batch encode documents for efficiency.
- Consider dimensionality reduction (PCA) if embedding dimensions are very high.
- Monitor index memory usage in production.

## Choosing the Right Index

Selecting the appropriate FAISS index depends on your requirements:

| Index Type | Dataset Size | Memory | Accuracy | Speed | Training |
|------------|-------------|--------|----------|-------|----------|
| `IndexFlatL2` | < 100K | High | Exact | Slow | No |
| `IndexIVFFlat` | 100K - 10M | High | Near-exact | Fast | Yes |
| `IndexIVFPQ` | 1M - 100M | Low | Approximate | Fast | Yes |
| `IndexHNSWFlat` | < 10M | Medium | Near-exact | Very fast | No |
| `IndexHNSWPQ` | 10M - 1B | Low | Approximate | Very fast | No |

### Decision Flowchart

1. **Do you need exact results?**
   - Yes → `IndexFlatL2` or `IndexFlatIP`
   - No → Continue

2. **Is memory constrained?**
   - Yes → `IndexIVFPQ` or `IndexHNSWPQ`
   - No → Continue

3. **Will you add vectors incrementally?**
   - Yes → `IndexHNSWFlat`
   - No → `IndexIVFFlat`

4. **Do you have GPU?**
   - Yes → Consider GPU variants for all above

## Performance Tuning

### Optimizing Search Speed

```python
# For IVF indexes, increase nprobe for better accuracy
index.nprobe = 20  # Default is often 1

# For HNSW, increase efSearch
index.hnsw.efSearch = 128

# Use multiple threads
faiss.omp_set_num_threads(8)
```

### Optimizing Memory

```python
# Use scalar quantizer for 4x memory reduction
index_sq = faiss.IndexScalarQuantizer(d, faiss.ScalarQuantizer.QT_8bit)
index_sq.add(vectors)

# Use PCA to reduce dimensions before indexing
pca_matrix = faiss.PCAMatrix(d, 64)  # Reduce to 64 dimensions
pca_matrix.train(vectors)
index_pca = faiss.IndexPreTransform(pca_matrix, faiss.IndexFlatL2(64))
```

### Benchmarking Your Index

```python
import time

# Warm-up
index.search(query, k)

# Benchmark
start = time.time()
for _ in range(100):
    index.search(queries, k)
elapsed = time.time() - start

qps = (100 * len(queries)) / elapsed
print(f"Queries per second: {qps:.0f}")
```

## Common Pitfalls and Solutions

| Problem | Cause | Solution |
|---------|-------|----------|
| `AssertionError: is_trained` | Called `add()` before `train()` on IVF/PQ | Call `train()` with representative vectors first |
| Slow search | Using exact index on large dataset | Switch to IVF or HNSW |
| High memory usage | Storing full-precision vectors | Use PQ or scalar quantization |
| Poor accuracy | `nprobe` or `efSearch` too low | Increase search-time parameters |
| Wrong similarity | Using L2 for cosine similarity | Normalize vectors and use inner product |
| Type errors | Not using `float32` | Ensure all arrays are `np.float32` |
